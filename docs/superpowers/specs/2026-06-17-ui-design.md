# Diseño — UI de escritorio (WPF) de la App de Resúmenes

Fecha: 2026-06-17
Sub-proyecto 1 de 3 (UI → Instalador → Pulido). El backbone funcional (Fases 1–4) ya está
operativo y verificado; esta UI es la cáscara visual de toda la V1.
Ubicación: todo dentro de `D:\Desarrollo\Programacion\Resumenes`.

---

## 1. Objetivo y alcance

Construir la **UI completa de la V1** en WPF, moderna ("Fluent" estilo Windows 11) y con
**foco máximo en mostrar el avance dentro de la app** (prioridad explícita del usuario).
Cubre: onboarding, inicio con historial, configuración del análisis, ejecución con progreso
rico, confirmación/edición de temas, resultados y configuración.

La consola `Resumenes.Cli` se conserva (headless/automatización). La UI NO la invoca: corre
el pipeline **in-process**.

**Definición de hecho:** los criterios de aceptación de la sección 7 se cumplen.

---

## 2. Decisiones (cerradas en brainstorming 2026-06-17)

- **Alcance:** V1 completa de entrada (8 pantallas, sección 4).
- **Look & feel:** Fluent moderno con **WPF-UI** (lepoco/wpfui, MIT): oscuro+claro, color de
  acento, navegación lateral, tarjetas, animaciones suaves, loader "pensante".
- **Ejecución in-process:** la UI referencia `Core`/`Infrastructure` y corre el pipeline en el
  mismo proceso, con progreso vía `IProgress<T>`.
- **Confirmación de temas con pausa interactiva:** Fase 1 → Fase 2 → la UI muestra los temas
  para editar/confirmar → Fase 3.
- **Validación visual:** capturas de la app real (no test automatizado de UI en v1).

## 3. Arquitectura

**Proyecto nuevo:** `Resumenes.Ui` (WPF, `net9.0-windows`; paquetes: `WPF-UI`,
`CommunityToolkit.Mvvm`). Referencia `Resumenes.Core` + `Resumenes.Infrastructure`.
MVVM estricto: Views (XAML) + ViewModels + servicios; sin lógica en code-behind.

**Refactor del backbone (mejoras justificadas para soportar la UI):**

1. **Reporte de progreso en el orquestador.** `PipelineOrquestador.EjecutarAsync` recibe un
   `IProgress<ProgresoPaso>` y emite eventos por etapa (iniciada/avance/completada/saltada/
   error) y sub-progreso real donde existe. La consola pasa un progress que loguea; la UI, uno
   que actualiza la pantalla.

2. **Sub-progreso real en los adaptadores:** `PaddleOcrServicio` reporta "página i/N" (ya
   recibe los resultados por página del worker); `PyMuPdfRasterizador` reporta el total de
   páginas; las etapas IA reportan "pensando" (indeterminado) + "chunk i/N" cuando corresponde.

3. **`ServicioAnalisis`** (en `Infrastructure`, usado por Cli y UI). Saca la orquestación de
   `Program.cs` y expone las 3 fases por separado:
   - `ProcesarArchivosAsync(carpeta, IProgress, ct)` → Fase 1 (limpieza por archivo) → archivos limpios.
   - `DetectarTemasAsync(analisis, promptTemas, ct)` → Fase 2 → `temas.json`.
   - `GenerarPorTemasAsync(analisis, temasConfirmados, IProgress, ct)` → Fase 3 (consolidar/resumir/PDF).
   El Cli las llama en secuencia; la UI inserta la pausa de confirmación entre Fase 2 y 3.

4. **Historial:** `IRepositorioEstado.ListarAnalisis()` (lectura) para la pantalla de inicio.

5. **Cancelación:** `CancellationToken` desde la UI a todo el pipeline (ya soportado).

`Core` queda como dominio puro; `Infrastructure` suma `ServicioAnalisis`; `Resumenes.Ui` es
solo presentación.

## 4. Mapa de pantallas y navegación

Shell con `NavigationView` (WPF-UI), rail lateral: **Inicio · Configuración · Ayuda**.
El análisis es una secuencia tipo *stepper* en el área principal.

1. **Onboarding (primer arranque)** — solo si falta config: bienvenida → descarga/verificación
   de dependencias (con progreso; se reusa en el sub-proyecto Instalador) → API key → carpeta
   de salida → Inicio. Si todo está, se saltea.
2. **Inicio** — título + fecha + saludo; botón "Nuevo análisis"; **historial** (tarjetas:
   nombre, fecha, estado, nº de temas; acciones: Ver resultados · Abrir carpeta · Continuar · Eliminar).
3. **Configurar análisis** — seleccionar carpeta de material (lista archivos + tipos), prompt
   opcional de temas del parcial, botón Analizar.
4. **Ejecutando** ⭐ — progreso en vivo de Fases 1–2 (sección 5).
5. **Confirmar temas** — temas detectados (nombre + archivos asignados); editar: renombrar,
   fusionar, quitar, reasignar; botón "Generar resúmenes" → Fase 3.
6. **Generando** — progreso en vivo de Fase 3 (mismo estilo).
7. **Resultados** — temas con su PDF; acciones: Abrir PDF · Abrir carpeta · Exportar (copiar
   todos); ruta de salida visible; reproceso manual (re-detectar temas / regenerar un tema).
8. **Configuración** — API key, carpeta salida, DPI, modelo, conservar intermedios, rutas
   runtime; tema claro/oscuro + acento.

Flujo: Inicio → Configurar → Ejecutando → Confirmar temas → Generando → Resultados → (Inicio,
queda en historial). Onboarding y Configuración son transversales.

## 5. Modelo de progreso "pensante" (pieza central)

**Señales reales del backbone:**
- Fases: 1 Limpieza · 2 Detección · 3 Resúmenes (stepper).
- Determinadas: archivos `i/N`, temas `i/N`, OCR páginas `i/N`, chunks `i/N`.
- Indeterminadas ("pensando"): cada llamada IA + conversión LibreOffice → loader pensante + estado.

**Evento:** `ProgresoPaso { fase, item, itemIndice, itemTotal, etapa, detalle, subIndice?,
subTotal?, estado }` (estado ∈ Iniciado/Avance/Completado/Error/Salteado).

**Pantalla "Ejecutando":**
- Arriba: nombre del análisis + **stepper de fases** (① Limpieza ② Temas ③ Resúmenes).
- Centro: **loader pensante** animado + estado cuando la IA piensa; **anillo/barra determinada**
  cuando hay número real (OCR páginas).
- **Lista de ítems** (archivos en Fase 1, temas en Fase 3) con chip de estado
  (pendiente/en proceso/✓/✗/saltado) + mini-progreso.
- Abajo: **barra global** + tiempo transcurrido + **Cancelar**.
- **Log en vivo** (panel colapsable) con timestamps.

**% global:** fases + conteos reales por ítem (el denominador de Fase 3 se afina al terminar la
detección); "pensando" indeterminado donde no hay número. Fiel a "avance real cuando se pueda".

**Cancelar/Reanudar:** Cancelar corta el token; lo hecho queda persistido. Reanudar muestra al
instante lo ya completado como ✓/saltado (idempotencia) y sigue desde donde quedó.

---

## 6. Datos/estado, errores y threading

- **Estado:** sin almacenamiento nuevo. Historial desde SQLite (`ListarAnalisis()`); editor de
  temas sobre `temas.json`; config en `settings.json`; API key en DPAPI.
- **Errores:** por-archivo/por-tema → chip ✗ + detalle, sin tumbar el resto (tolerancia ya en
  el servicio). Entorno (faltan deps / sin key / sin internet) → mensaje accionable + ruta a
  onboarding/configuración. Excepción no controlada → pantalla de error amable + log; nunca
  crash mudo. Cancelar → corte limpio + estado parcial persistido.
- **Threading:** pipeline en background (async/`Task.Run`); progreso marshaleado al hilo de UI
  con `IProgress<T>`; la ventana no se congela.

## 7. Criterios de aceptación

1. Primer arranque sin config → onboarding guía a cargar API key + carpeta (las deps se
   abordan en el sub-proyecto Instalador; en dev se asume el runtime ya presente).
2. Seleccionar una carpeta con material → Ejecutando muestra fases, archivos `i/N`, OCR
   páginas `i/N` y loader pensante en las etapas IA, con log en vivo, sin congelarse.
3. Tras la detección, la pantalla Confirmar temas permite renombrar/fusionar/quitar/reasignar;
   al confirmar, se generan los PDFs por tema.
4. Resultados lista los temas con sus PDFs y permite Abrir PDF / Abrir carpeta / Exportar.
5. Un archivo que falla aparece con ✗ y su motivo, y el resto del lote igual termina.
6. Cancelar detiene sin crashear; reabrir el mismo análisis reanuda mostrando lo ya hecho.
7. Inicio lista los análisis previos con su estado y acciones.
8. Tema claro/oscuro + acento aplicables desde Configuración.

## 8. Fuera de alcance (de esta UI v1)

Descarga real de dependencias (es el sub-proyecto **Instalador**; acá solo el lugar/onboarding
donde irá); test automatizado de UI; edición manual de los textos limpios/resúmenes (la V1 lo
mencionaba como deseable — se puede sumar luego); multi-idioma de la interfaz.

---

## 9. Riesgos y mitigaciones

- **Refactor de progreso toca el orquestador (16 tests):** mantener firma compatible
  (`IProgress` opcional con default no-op) para no romper la consola ni los tests.
- **WPF-UI/animaciones:** usar los controles de la librería; el loader pensante como control
  reutilizable; evitar animaciones que traben el hilo de UI.
- **Sub-progreso de OCR:** requiere exponer el "página i/N" desde `PaddleOcrServicio`; si no se
  logra fino, degradar a "OCR en proceso" (indeterminado) sin bloquear el resto.
